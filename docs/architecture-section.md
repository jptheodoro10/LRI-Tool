# Architecture

The application was implemented as a web-based system composed of a React/Vite frontend, a FastAPI backend, a PostgreSQL database, and an abstraction layer for language-model services. The backend follows a layered organization, structured as `Router -> Service -> Repository -> Database`. This design separates HTTP concerns from business rules and persistence logic, improving maintainability, testability, and the ability to evolve individual modules with limited ripple effects.

At the domain level, the central aggregate is `Run`, which represents a single Lean Research Inception execution. A `Run` stores project metadata, current phase, current cycle, execution status, and the final problem synthesis. The workflow is organized into five phases: initial canvas completion, alignment and participant invitation, problem review, semantic differential assessment, and final decision with PDF export. In addition, the system supports iterative reformulation through cycles. When a final decision is `PIVOT`, the application returns the run to Phase 2, increments the cycle counter, and reuses the most recent valid canvas state as the baseline for the next iteration. This mechanism preserves continuity across iterations and reduces rework during problem reformulation.

The frontend provides the facilitator-facing interface for canvas editing, phase progression, invite generation, result inspection, and AI-assisted interactions. The backend exposes REST endpoints for authentication, project management, canvas operations, invite handling, and scoring. Access control distinguishes two usage modes: facilitators authenticated with JWT tokens and invited participants authenticated through public invite links plus participant identifiers. This split lowers the entry barrier for external participants while keeping authorization checks explicit in the backend.

The AI layer is implemented as a cross-cutting capability behind a single `LLMClient.generate(prompt)` interface. Two implementations are currently available: a deterministic mock client for offline and reproducible execution, and an OpenAI-based client using the Responses API. This abstraction decouples the core application logic from a specific provider SDK and simplifies experimentation, provider replacement, and local testing.

Two AI-supported workflows are exposed to users. In Phase 1, the system generates recommendations for empty canvas fields using the content of already filled fields as context. In Phase 3, it generates analytical overviews for each canvas field once the board is complete. Although both workflows rely on the same prompt-generation and provider abstraction, they serve different goals: Phase 1 assists completion, whereas Phase 3 supports reflection and refinement.

A relevant architectural decision was to issue AI requests in parallel at the frontend, one request per target field. When the facilitator requests Phase 1 recommendations or Phase 3 overviews, the interface dispatches multiple independent calls and renders results progressively as soon as each response returns. This strategy reduces perceived latency because users do not need to wait for the slowest field before seeing initial results. In practice, it produces a more responsive interaction pattern without requiring a more complex asynchronous job infrastructure.

Persistence decisions differ across the two AI workflows. Phase 1 recommendations are stored in the database together with execution status, timestamps, and a `context_hash` derived from the relevant canvas state and prompt version. This allows the system to avoid unnecessary recomputation when the contextual input has not changed, reducing cost and response time. In contrast, Phase 3 overviews are intentionally transient and remain only in frontend state. Since these overviews act as reflective support rather than canonical project data, not persisting them keeps the database focused on durable workshop artifacts.

The collaboration model also reflects deliberate simplification choices. The canvas is reused across Phases 1 to 3, but direct editing is restricted to the facilitator. Invited participants contribute mainly in Phase 4 through semantic differential scoring. This reduces concurrent-write conflicts and preserves consistency in the primary artifact under construction. During Phase 2, the facilitator generates tokenized invite links. In the first cycle, at least one invite must be generated before moving to Phase 3; after a pivot, new invites are locked and the same participant set is reused. This decision stabilizes the evaluation group across reformulation cycles.

Finally, the system consolidates workshop outcomes in Phase 5. Participants submit 1-to-7 ratings for three metrics, and the backend computes aggregates such as distributions, averages, and medians. After a final `GO` or `ABORT` decision, the application exports a PDF report containing the formulated problem, assessment summaries, and the facilitator's synthesis. Overall, the architecture prioritizes local reproducibility, clear separation of concerns, and responsive AI-assisted interaction, while leaving room for future evolution such as real background workers and stronger observability mechanisms.
